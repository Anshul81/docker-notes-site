Why Docker Exists

The "it works on my machine" crisis

Imagine spending two weeks building a Python web app. It works perfectly on your laptop. You send the code to a teammate. They run it. It crashes. You look at the error together and say the four most dreaded words in software development: "it works on my machine."

This isn't a rare edge case вҖ” before Docker, it was the norm. The code was identical. The environment was different: your laptop had Python 3.10, theirs had 3.7. You had requests==2.28, they had 2.20. You were on Windows, the server runs Linux. You had an environment variable set that they didn't.

How people tried to solve this, historically

Era 1 вҖ” the manual README (1990sвҖ“2000s)

SETUP INSTRUCTIONS:
1. Install Python 3.7 (NOT 3.8, it breaks)
2. Install these 14 libraries (exact versions listed)
3. Set these 6 environment variables
4. Make sure your OS locale is set to UTF-8
5. Pray

Setup took hours. One wrong step meant a broken environment. Onboarding a new teammate could eat an entire day.

Era 2 вҖ” Virtual Machines (2000sвҖ“2010s)

The next idea: ship the entire computer. A VM is a software-simulated computer running inside your real one вҖ” package the full OS plus your app, and everyone runs an identical environment.

This genuinely worked вҖ” but at a real cost:

  • A single VM is typically 5вҖ“20 GB
  • Booting takes 2вҖ“5 minutes
  • Running 10 services means 10 full VMs вҖ” your machine grinds to a halt
  • Each VM carries a full OS even if the app itself is only 10MB
  • Hardware resources are reserved upfront, whether the app needs them right now or not

The shipping container analogy

Docker's name isn't random. Before standardized shipping containers existed (pre-1950s), loading cargo onto ships was chaos вҖ” every item was different. Bananas needed special handling, machinery needed cranes, glass needed padding. Every port had different equipment for different cargo.

In 1956, the standardized shipping container was invented. Same size, same corners, same locking mechanism вҖ” every ship, every port, every truck could handle it, regardless of what was inside.

Docker does the exact same thing for software. Your app, its dependencies, and its config get packaged into a standard container. It runs identically on your laptop, your colleague's machine, the test server, and production.

What a container actually is

mental model A container is an isolated process running on your machine that thinks it's the only thing running on its own private computer.

It has its own:

  • Filesystem (can't see your files)
  • Network (has its own IP)
  • Process list (can't see other processes)
  • Resource limits (gets only the CPU/RAM assigned to it)

But it shares the host OS kernel вҖ” that's the entire reason it's lightweight. Containers start in milliseconds, not minutes, and measure in megabytes, not gigabytes.

Why Docker specifically?

Linux containers as a raw technology existed before Docker вҖ” but using them required deep kernel expertise. Docker made it simple with a standard image format, a straightforward CLI (docker run, docker build), a public registry to share images (Docker Hub), and later a standard way to describe multi-container infrastructure (Compose). Docker didn't invent containers вҖ” it democratized them.

Real industry usage

  • Netflix runs thousands of microservices in containers вҖ” recommendations, billing, search, each isolated.
  • CI/CD pipelines (like GitHub Actions) run your tests inside a fresh container every single time.
  • ML engineers package a model + Python + CUDA dependencies into one container and share it with anyone.
  • You, after this course вҖ” shipping a Flask + PostgreSQL + Redis stack to any server with one command.

Installing on Windows вҖ” why WSL2 matters

Docker containers are fundamentally Linux technology. They depend on two Linux kernel features:

  • Namespaces вҖ” isolation (a container thinks it's alone)
  • Cgroups вҖ” resource limits (CPU/RAM per container)

Neither exists in the Windows kernel. So on Windows, something has to provide a real Linux kernel underneath. Docker Desktop uses WSL2 вҖ” a lightweight Linux VM built into Windows вҖ” to do this transparently. You never see or manage it directly; your containers actually run inside it.

Image vs. container вҖ” proven with real commands

Running docker run pulls an image (the blueprint) and creates a container (a running instance of it). Removing a container never removes the image вҖ” only the instance disappears. 

docker run -d -p 8080:80 --name my-nginx nginx
docker stop my-nginx
docker rm my-nginx

docker images   # nginx is still here вҖ” the blueprint survives
docker ps -a    # container is gone вҖ” the instance does not

Real terminal evidence from this exact test: 

CONTAINER ID   IMAGE         STATUS                        NAMES
0ab4cf6d5bc4   nginx         Exited (137) 32 seconds ago   nostalgic_carver
4c69dd85f64e   hello-world   Exited (0) 4 minutes ago      inspiring_germain
reading exit codes Exited (0) means the process finished on its own, successfully вҖ” that's hello-world, which just prints a message and ends. Exited (137) means the process was killed by an external signal вҖ” that's nginx, stopped deliberately via docker stop. This distinction becomes a genuine debugging clue later: when a container crashes unexpectedly, the exit code is often the first thing worth checking.

Also worth noticing: Docker auto-generated the names nostalgic_carver and inspiring_germain because none were specified. In real projects you'll always name containers explicitly with --name вҖ” it makes logs, debugging, and networking (Chapter 6) far easier to reason about.
