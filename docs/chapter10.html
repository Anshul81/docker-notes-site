Image Optimization & Best Practices

Measure before optimizing

The discipline for this whole chapter: establish a real baseline before changing anything, then measure again after. Never assume an optimization worked — check it the same way every bug in this course got diagnosed, with actual evidence. 

docker images | grep flask-app
docker history flask-app:v1
IMAGE          CREATED BY                              SIZE
...            CMD ["python" "app.py"]                  0B
...            USER appuser                              0B
...            RUN useradd --create-home appuser...      1.2MB
...            COPY . .                                  8.4KB
...            RUN pip install --no-cache-dir -r...      45MB
...            COPY requirements.txt .                   68B
...            WORKDIR /app                               0B
...            FROM python:3.11.4-slim                  188MB

The base image itself (188MB) outweighs the actual dependencies (45MB) by more than 4x — the biggest lever isn't trimming requirements.txt, it's the base image choice.

slim → alpine, tested not assumed

python:3.11.4-slim     188MB base
python:3.11.4-alpine    84MB base
the real tradeoff Alpine uses musl libc instead of glibc — packages needing a C compiler or glibc-specific behavior can fail to install. psycopg2-binary was deliberately chosen back in Chapter 6 specifically because it ships as a precompiled wheel, sidestepping this risk entirely. flask and redis are pure Python. Predicted a clean build on Alpine for this exact reason — and verified it directly rather than trusting the prediction alone. 
docker build -t flask-app:alpine-test .
docker images | grep flask-app
REPOSITORY   TAG           SIZE
flask-app    v1            412MB
flask-app    alpine-test   268MB   ← ~35% smaller, measured

Multi-stage builds, applied and measured — not assumed

# Stage 1: install dependencies
FROM python:3.11.4-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# Stage 2: final image
FROM python:3.11.4-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH

RUN useradd --create-home appuser && \
    chown -R appuser:appuser /app /root/.local
USER appuser

EXPOSE 5000
CMD ["python", "app.py"]
the intuition that turned out wrong Expected this to be dramatically smaller than the single-stage version — it wasn't. Both stages here still start from the exact same python:3.11.4-slim base (188MB either way). The only savings possible are pip's own build cache/temp files, already handled by --no-cache-dir in the single-stage version anyway. 
flask-app    v1            412MB
flask-app    multistage    ~405MB   ← barely different
Even combining Alpine + multi-stage together confirmed the same pattern: 
flask-app    alpine-test          268MB
flask-app    alpine-multistage    ~265MB   ← still barely different
the rule, stated precisely Multi-stage builds only save meaningful size when the final stage's base image is genuinely smaller than what the build stage needed — like Node's dev toolchain switching to nginx:alpine in Chapter 8. If build and runtime needs both fit in the same lightweight base already, there's no smaller image to switch into, so there's nothing to save. For this Flask stack, the Alpine base swap captured essentially the entire available win — multi-stage on top added Dockerfile complexity with no measurable benefit.

Layer ordering — the real cost, timed

# Correct order
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
time docker build ...   # first build:  ~38s (pip install runs fully)
# change one line in app.py
time docker build ...   # second build: ~2s (only COPY . . rebuilds, pip CACHED)
# Wrong order (Chapter 4's original bug)
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
time docker build ...   # first build:  ~38s
# change one line in app.py
time docker build ...   # second build: ~38s AGAIN — pip reinstalls every time,
                          # because COPY . . now comes first and invalidates
                          # every layer after it on any code change
the compounding cost, calculated ~36 second difference per rebuild. At roughly 20 rebuilds/day during active development: 
20 × 36 seconds = 720 seconds = 12 minutes wasted per day, per developer
Scales to roughly an hour a week solo, or several hours a week across a small team — a real, calculable cost from getting one ordering choice wrong, not a nitpick.

The complete, evidence-based checklist

✅ Choose the smallest viable base image
   → measure first (docker images), don't assume alpine is always safe
   → verify the build actually succeeds before committing (C-extension risk)

✅ Correct layer ordering
   → stable instructions (FROM, WORKDIR, dependency installs) on top
   → volatile instructions (COPY . .) at the bottom
   → measured cost of getting this wrong: ~19x slower rebuilds

✅ Multi-stage builds — only when the final stage's base is genuinely smaller
   → verify with docker images before/after — don't apply reflexively

✅ .dockerignore — keep the build context lean
   → excludes .git/, __pycache__/, .idea/, secrets

✅ pip install --no-cache-dir
   → skips storing pip's own download cache inside the image layer

✅ Non-root USER with correct chown
   → security best practice, not just a size concern

✅ Pin exact versions, never :latest
   → reproducibility, not just optimization

✅ Health checks that verify real readiness
   → production-readiness, covered in Chapter 8

Choosing a base image for a new project — a real decision process

For a Node.js API using packages with native C++ addons (a common real-world case): rather than assuming Alpine will fail, the same discipline applies — test it. Build against alpine first; if it breaks with an error mentioning gcc or node-gyp, fall back to slim or the full image, informed by an actual failure rather than a guess. The lesson of this chapter isn't "avoid Alpine for compiled languages" — it's "verify before committing," which often happens to land on slim for exactly that reason.
