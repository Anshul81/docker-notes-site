Networking in Docker

The problem: two isolated islands

A Flask container and a PostgreSQL container are two completely separate, namespaced processes (Chapter 2) — no shared filesystem, no shared process list. Manually running SQL via docker exec every time isn't how real applications talk to a database. A real app needs to connect to it over the network, like it would with any external database server.

The tempting but wrong first guess

The instinct is often "they need connected ports." Testing that against something already known: -p 5000:5000 was used back in Chapter 1 to open localhost:8080 in a browser. Who was on the other end of that — another container, or the browser on the host machine?

the actual answer -p connects your laptop to a container. It has nothing to do with letting two containers reach each other — that's a completely separate mechanism.

The default bridge network's real limitation

Every container attaches to Docker's default bridge network unless told otherwise: 

docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# xxxxxxxxxxxx   bridge    bridge    local
# xxxxxxxxxxxx   host      host      local
# xxxxxxxxxxxx   none      null      local

On the default bridge, containers can only reach each other by IP address — name-based DNS resolution simply doesn't work there (a real, documented Docker limitation, kept mainly for backward compatibility). Since internal IPs are reassigned on every restart, hardcoding one into an app's config is fragile and will eventually break. 

Default bridge network:
  flask-app tries: my-postgres        ❌ fails, no DNS resolution
  flask-app tries: 10.0.1.3           ✅ works, but IP changes on restart

The fix — user-defined networks

docker network create my-app-network

docker run -d --name my-postgres --network my-app-network \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

docker run -d --name flask-app --network my-app-network \
  -p 5000:5000 \
  flask-app:v1

On a user-defined network, Docker provides working built-in DNS — any container can reach another by its container name, and that resolution stays correct even after a restart changes the underlying IP. 

User-defined network:
  flask-app tries: my-postgres    ✅ works! Docker resolves it automatically
                                      to whatever IP it currently has

Proven two independent ways, not assumed

docker exec -it flask-app bash
python3 -c "import socket; print(socket.gethostbyname('my-postgres'))"
# → 172.18.0.2
docker network inspect my-app-network
"Containers": {
    "2071cdf6...": {
        "Name": "my-postgres",
        "IPv4Address": "172.18.0.2/16"
    },
    "98be598e...": {
        "Name": "flask-app",
        "IPv4Address": "172.18.0.3/16"
    }
}

Two completely independent sources — Python's own DNS lookup, and Docker's own network records — agreeing on the exact same IP. That's real proof, not a lucky coincidence.

Real connection code

DB_HOST = os.environ.get('DB_HOST', 'my-postgres')  # container name, not IP

Same ENV-as-default pattern from Chapter 4, now applied to networking — the container name is stable and overridable, unlike a hardcoded IP that would break on the next restart.

Wiring it into Flask for real

import psycopg2

@app.route('/students')
def get_students():
    conn = psycopg2.connect(
        host=os.environ.get('DB_HOST', 'my-postgres'),
        database='postgres',
        user='postgres',
        password=os.environ.get('DB_PASSWORD', 'mysecretpassword')
    )
    cur = conn.cursor()
    cur.execute('SELECT id, name FROM students;')
    rows = cur.fetchall()
    cur.close()
    conn.close()
    students = [{"id": r[0], "name": r[1]} for r in rows]
    return {"students": students}
debugging detour: "relation students does not exist" — again Hit the exact same error message as Chapter 5's typo bug — but this time the volume path was verified correct from the start. That ruled out a repeat of the earlier mistake, so the investigation went further:
  1. Checked docker logs my-postgres — found: "PostgreSQL Database directory appears to contain a database; Skipping initialization" — meaning Postgres genuinely loaded the existing volume, not a fresh one.
  2. Compared the volume's CreatedAt timestamp against Postgres's own crash-recovery log line ("database system was interrupted; last known up at ...") — the timestamps lined up, initially suggesting continuity with the earlier working session.
  3. Checked the actual data directory size on disk: du -sh /var/lib/postgresql/data → 39M — the size of a freshly-initialized, genuinely empty Postgres data directory (just internal system catalogs and WAL logs), not one that had ever held a real table.
Conclusion: the pgdata volume had, at some point between sessions, been wiped and recreated fresh — most likely during earlier cleanup commands (docker volume rm pgdata / prune) run while fixing the Chapter 5 typo bug. The networking mechanism itself was never broken — DNS resolution had already been proven correct, independently, before this error even showed up. Recreated the table, and the full chain worked end to end: 
{"students": [{"id": 1, "name": "Anshul"}]}
Browser → Flask (via -p) → DNS-resolved my-postgres → real Postgres query → JSON back out.
