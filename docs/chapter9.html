Debugging, Logging & Troubleshooting

The debugging mindset — formalizing what already worked

Chapters 5, 6, and 7 each involved real bugs (a volume path typo, a "students table missing" mystery, a stale image cache) — and in every case, the first move was the same: docker logs. This chapter turns that instinct into a repeatable, deliberate method.

why logs come first, always The outside world only ever sees the symptom — a browser showing "Internal Server Error" is generic and unhelpful. The container's own logs contain the actual cause: the exact traceback, the exact line, the exact exception.

The systematic debugging ladder

1. docker ps -a
   → Is the container even running? What's its exit code?

2. docker logs <container>
   → What does the application itself say went wrong?

3. docker inspect <container>
   → Mounts correct? Environment variables set as expected?
     Network attached correctly?

4. docker exec -it <container> bash
   → Get inside. Check files exist. Check permissions.
     Try running the failing command manually.

5. docker stats
   → Is it a resource problem? Maxed CPU/memory?

Every one of these tools was already used organically in earlier chapters — this is about recognizing them as a deliberate ladder, not separate unrelated tricks.

Exit codes — a real reference

CodeMeaning
0Success — process finished normally
1General application error (unhandled exception, etc.)
2Tool-specific error (e.g. Python's own "bad file/argument" convention)
126Command found but not executable (permissions issue)
127Command not found (typo in CMD, or tool not installed)
137SIGKILL (128+9) — killed forcibly, e.g. docker stop, or OOM kill
143SIGTERM (128+15) — graceful stop signal received
build-time vs. runtime failure A genuinely malformed Dockerfile (like Chapter 4's squashed multi-instruction line) fails at docker build time — you never even get an exit code, because the image never finishes building. Exit code 127 is different: it means the container started successfully, and the command Docker tried to run inside it simply doesn't exist there. Different failure stages entirely — recognizing which one you're in immediately narrows down where to look.

Debugging exercise — broken Dockerfile #1

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "main.py"]   # actual file is named app.py
the subtle distinction: 127 vs. 2 This is not exit 127. python itself exists in the image — it's found and runs successfully. It's main.py that doesn't exist, and Python itself reports that error, not the shell. 
docker run myapp
# python3: can't open file '/app/main.py': [Errno 2] No such file or directory

docker ps -a
# STATUS: Exited (2)
Exit code 2 here is Python's own convention for "I ran, but hit an error with my arguments/file" — a different category from "command not found." Different tools have their own exit code conventions beyond the universal ones — the code narrows the category, but docker logs still tells you the specific fact.

Debugging exercise — broken Dockerfile #2 (the silent one)

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 5000
CMD ["python", "app.py"]

Builds fine. docker run -d -p 5000:5000 myapp shows Up 3 seconds — genuinely running, no crash. But http://localhost:5000 gives ERR_CONNECTION_REFUSED — the browser can't even reach it.

a bug with zero error output, anywhere Cause: app.py still has app.run(host='127.0.0.1', ...) instead of 0.0.0.0. Flask binds correctly and listens — but only to itself. From outside the container (via -p), that's a refused connection. 
docker logs myapp
# * Running on http://127.0.0.1:5000
# (no errors — Flask genuinely believes everything is fine)
This is a category of bug that produces zero error output anywhere, which makes it harder to debug than a crash — the standard "check docker logs" instinct comes up completely empty. The next diagnostic step is docker exec, verifying directly from inside: 
docker exec -it myapp bash
curl http://127.0.0.1:5000   # works — proves Flask IS running,
                              # just only reachable from inside
Confirming it works from inside but not outside narrows the cause specifically to a binding issue, not a crash or missing dependency.
the real lesson Not every bug produces an error. Sometimes the application is working exactly as coded — the code itself is just wrong in a way that only shows up as silence. When logs are clean but something's still broken, verify behavior directly from inside the container rather than continuing to stare at logs that were never going to show anything.

Logging — how docker logs actually works

docker logs <container>              # full log history, printed once
docker logs -f <container>           # follow mode — live streaming
docker logs --tail 50 <container>    # only the last 50 lines
docker logs --since 10m <container>  # only logs from the last 10 minutes
docker logs -t <container>           # add timestamps to each line

For a container running for days where something broke minutes ago, --since 10m cuts straight to the relevant window instead of dumping days of irrelevant noise.

where logs actually come from Docker doesn't read your source code. It captures whatever a container's main process writes to stdout and stderr — the same two universal output streams every Linux process has always had. This is why docker logs works identically for Python, Node.js, Nginx, or anything else — it doesn't care what language or framework is involved, only that output goes to those two streams.

Why PYTHONUNBUFFERED=1 actually mattered (Chapter 4, revisited)

Python buffers output by default when not connected to an interactive terminal — it collects print() statements in memory and only flushes them in batches. 

Without PYTHONUNBUFFERED=1:
  print("Processing request...")
  → sits in a memory buffer, doesn't reach stdout immediately
  → docker logs -f shows NOTHING, or shows it minutes late

With PYTHONUNBUFFERED=1:
  print("Processing request...")
  → flushes to stdout immediately
  → docker logs -f shows it in real time, as it happens
why this is a nasty class of bug Without it, you could be watching docker logs -f live while a bug is actively happening, and see nothing — not because logging is broken, but because output is sitting in an unflushed buffer. It creates the illusion that your primary debugging tool has stopped working, when really a single missing environment variable is the cause.

Debugging a "silent" container — the full method combined

Symptom: Flask container shows "Up 10 minutes",
         but the app is unresponsive. docker logs shows almost nothing.

1. docker ps -a           → confirm it's still running, check exit code if not
2. docker logs --since 5m → check recent activity specifically
3. docker stats           → CPU pinned at 100%? Memory maxed?
                             (a stuck infinite loop often shows as high CPU
                              with zero new log output)
4. docker exec -it <container> bash
   → get inside, check if the process is actually responsive
   → try curling the app FROM INSIDE its own container
docker stats
CONTAINER   CPU %   MEM USAGE / LIMIT   MEM %
flask-app   98.2%   340MiB / 512MiB     66.4%

High CPU with no corresponding log output is a strong signal of an infinite loop or a stuck blocking call (e.g. a database connection attempt that hangs forever waiting for a database that never responds) — a distinct failure signature from a crash, and one docker logs alone wouldn't surface clearly.
