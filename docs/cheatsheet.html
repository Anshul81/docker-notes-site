Command Cheat Sheet

Containers

docker run -d -p 8080:80 --name web nginx create + start, detached, port mapped, named docker ps / docker ps -a running only / all containers docker start / stop / rm <name> lifecycle control docker exec -it <name> bash enter a running container docker logs <name> always check this first when debugging

Images

docker build -t name:tag . build from Dockerfile in current dir docker images / docker rmi <name> list / remove docker tag src:tag new/name:tag pointer only, no data copied docker push / docker pull registry round-trip docker history <image> inspect layers

Volumes

docker volume create mydata Docker-managed persistent storage -v mydata:/path/in/container named volume mount -v C:/host/path:/path/in/container bind mount — exact host path docker volume ls / inspect / rm / prune manage volumes

Networking

docker network create my-net user-defined = working DNS by name --network my-net attach a container at run time docker network ls / inspect <net> list / see members + IPs -p host:container host↔container only — NOT container↔container

Dockerfile instructions

FROM image:tag always first, pin exact version WORKDIR / COPY / RUN build-time — baked into layers ENV KEY=default overridable at runtime with -e CMD ["python", "app.py"] runtime default, fully replaceable USER appuser chown -R first, or user can't write its own files

Compose

docker compose up / up -d start the whole stack docker compose up --build use after ANY code/Dockerfile change docker compose down / down -v keep volumes / also delete volumes docker compose logs -f <service> follow one service's logs depends_on: condition: service_healthy wait for real readiness, not just "started"
quick decision rules
  • Named volume vs. bind mount: database data → named volume. Live code editing → bind mount.
  • ENV vs. ARG: needed at runtime → ENV. Only needed during build → ARG.
  • RUN vs. CMD: should happen once, at build time → RUN. Should happen every container start → CMD.
  • Forgot to rebuild? If code changed and behavior looks stale, check whether you ran --build.
  • Container-to-container not working? Confirm both are on the same user-defined network, and you're using the service/container name, not an IP.
