Volumes & Persistent Storage

Containers forget everything

docker rm deletes the container's entire writable layer. Anything written there — logs, uploads, database rows — is gone permanently. This is intentional: containers are meant to be disposable.

mental model A container is a hotel room — checkout resets everything. A volume is a storage locker outside the hotel — it survives no matter how many rooms (containers) you check in and out of.

Three types of storage

TypeLocation decided byBest for
Named volumeDockerDatabases, persistent app state
Bind mountYou (exact host path)Local dev, live code editing
tmpfsRAM onlyShort-lived secrets, never touches disk

Named volumes

docker volume create mydata
docker run -it -v mydata:/app/data --name test ubuntu bash

Data written to /app/data is written to the volume — completely decoupled from any single container's lifetime. Proved this directly: wrote a file in one container, removed it entirely, started a brand-new container on the same volume, and the file was still there.

Bind mounts

docker run -d -v C:/path/to/project:/app -p 5000:5000 flask-app:v1

Maps an exact host folder into the container. Edits on the host show up inside the container instantly — no rebuild needed. But a bind mount doesn't merge with the image's files at that path, it completely overlays them. An older file on the host will hide a newer one baked into the image.

real bug: the postgresql path typo Mounted a volume to /var/lib/postgres/data (typo — missing "ql"). PostgreSQL actually writes to /var/lib/postgresql/data. Docker didn't error — it just mounted exactly the (wrong) path given, and silently created an anonymous volume for the real path Postgres needed. Result: data appeared to "vanish" after container recreation, because it was never in the named volume to begin with.

Fix: always verify the mount immediately with docker inspect <container> --format "{{.Mounts}}" before trusting it — and get exact internal data paths from the image's official Docker Hub page, never from memory.

Common data paths (from official images)

postgres   →  /var/lib/postgresql/data
mysql      →  /var/lib/mysql
mongodb    →  /data/db
redis      →  /data

No consistent pattern across images — each application decides its own path. Docker has no awareness of what any given app expects; it just mounts whatever path you tell it to, correct or not.

Useful commands

docker volume ls
docker volume inspect pgdata
docker volume rm pgdata
docker volume prune          # remove all unused volumes

Backing up a named volume's data

docker run --rm -v pgdata:/data -v C:/backup:/backup ubuntu \
  tar czf /backup/pgdata-backup.tar.gz /data

A throwaway container mounts both the named volume and a host folder, just to copy data between them.
