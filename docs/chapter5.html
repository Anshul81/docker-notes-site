chapter 05 / 07

Volumes & Persistent Storage

Containers forget everything

Predicting the outcome of a real sequence, before running it:

docker exec -it api-test bash
echo "important data" > /app/mynotes.txt
exit

docker stop api-test
docker rm api-test

docker run -d -p 5000:5000 --name api-test docker-api:v1
docker exec -it api-test bash
cat /app/mynotes.txt   # ?

The file is gone. docker rm doesn't just stop a container — it deletes the entire writable layer. Anything written there (logs, uploads, database rows) disappears permanently. 

Container writable layer
┌─────────────────────────────┐
│ mynotes.txt  ← lives HERE   │  ← deleted when container is removed
├─────────────────────────────┤
│ Image layers (read-only)    │  ← untouched, reusable
└─────────────────────────────┘

This is intentional, not a bug — Docker's design philosophy is that containers should be disposable. Great for a stateless web server. Genuinely dangerous for a database: if a teammate accidentally runs docker rm -f postgres-container on a plain, volume-less setup, months of customer data are gone forever — no recycle bin, no undo.

The solution — volumes

analogy A container is a hotel room — checkout resets everything. A volume is a storage locker outside the hotel — it survives no matter how many rooms (containers) you check in and out of. 
Without volume:                    With volume:

Container                          Container
┌───────────────┐                  ┌───────────────┐
│ Writable Layer│                  │ Writable Layer│
│ data.db  ←────┼─ dies with       │  (points to)  │
└───────────────┘  container       └───────┬───────┘
                                            │
                                            ▼
                                    ┌───────────────┐
                                    │ Docker Volume │ ← lives on host,
                                    │ data.db       │   survives container
                                    └───────────────┘   removal

Three types of storage

TypeLocation decided byBest for
Named volumeDockerDatabases, persistent app state
Bind mountYou (exact host path)Local dev, live code editing
tmpfsRAM onlyShort-lived secrets, never touches disk

Named volumes, proven directly

docker volume create mydata
docker run -it -v mydata:/app/data --name vol-test ubuntu bash

# inside:
echo "This data should survive" > /app/data/notes.txt
cat /app/data/notes.txt
exit

docker rm vol-test    # container fully gone

docker run -it -v mydata:/app/data --name vol-test-2 ubuntu bash
# inside:
cat /app/data/notes.txt   # still there

vol-test and vol-test-2 are completely different containers — different IDs, different writable layers — yet notes.txt survives, because the volume never depended on either container's existence. Breaking down the flag: 

-v  mydata          :         /app/data
     │                              │
     │                              └─ path INSIDE the container
     └─ volume name (Docker manages where this actually lives)

Listing and inspecting volumes

docker volume ls
docker volume inspect mydata
docker volume rm mydata
docker volume prune          # remove all unused volumes
[
    {
        "Name": "mydata",
        "Mountpoint": "/var/lib/docker/volumes/mydata/_data",
        "Driver": "local"
    }
]

That Mountpoint is the real location inside the WSL2 Linux VM on Windows — you rarely touch it directly, but knowing it exists helps when debugging.

Bind mounts — the other type

Named volumes suit databases where the exact physical location doesn't matter. Bind mounts solve a different problem: actively editing code and wanting the container to reflect every change instantly, without rebuilding. 

docker run -d -v C:/Users/anshu/docker-learning/chapter4:/app -p 5000:5000 flask-app:v1
-v  C:/Users/anshu/.../chapter4  :  /app
     │                                   │
     │                                   └─ path inside container
     └─ EXACT path on YOUR machine (you choose it, not Docker)
Named VolumeBind Mount
LocationDocker decidesYou decide (exact path)
Managed byDockerYou, manually
Best forDatabases, persistent stateLocal dev, live editing
PortabilityVery portableTied to your machine's folder layout
verified live Bind-mounted a Windows folder over a container's /app, edited app.py on the host, and — without rebuilding — refreshed the browser. The new text appeared immediately. A bind mount doesn't merge with what's underneath in the image; it completely overlays that path. Whatever was baked into the image at /app becomes invisible the moment something else is mounted on top of the same path.
the gotcha this creates If the image was built with COPY . . baking in a newer app.py, but the bind-mounted host folder has an older version — the older, host version wins, completely hiding the newer one baked into the image. Also: pip install already ran during build, before the mount ever existed. Adding a new package to requirements.txt on the host and expecting it to "just work" via the live mount doesn't — the mount only solves "my code changed," not "my dependencies changed." A new dependency still requires a real rebuild.

Mini Project 2 — PostgreSQL with a persistent volume, and a real bug

docker volume create pgdata

docker run -d \
  --name my-postgres \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -v pgdata:/var/lib/postgres/data \
  -p 5432:5432 \
  postgres:16

Created a table, inserted a row, destroyed the container, recreated from the same volume, and checked:

CREATE TABLE students (id SERIAL PRIMARY KEY, name TEXT);
INSERT INTO students (name) VALUES ('Anshul');
SELECT * FROM students;
-- destroy, recreate container...
SELECT * FROM students;
-- ERROR:  relation "students" does not exist
the actual bug: a one-character path typo Debugged systematically rather than guessing — checked docker volume ls, docker volume inspect pgdata, and critically docker inspect <container> --format "{{.Mounts}}", which revealed two mount entries instead of one: 
Mount 1:  volume pgdata     →  /var/lib/postgres/data      ← what was typed
Mount 2:  volume e6ba24...  →  /var/lib/postgresql/data    ← what Postgres actually uses
/var/lib/postgres/data (typed) vs. /var/lib/postgresql/data (real Postgres data path — note the extra "ql"). Docker didn't error — it has no idea what path any given application expects internally. It simply mounted pgdata to the exact (wrong) path given, and since PostgreSQL's real data path had no volume attached, Docker silently created an anonymous volume there instead. The students table was written into that anonymous volume the whole time — not into pgdata.
the fix, and the habit it created 
docker stop my-postgres-2 && docker rm my-postgres-2
docker volume rm pgdata && docker volume prune
docker volume create pgdata

docker run -d --name my-postgres \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -v pgdata:/var/lib/postgresql/data \
  -p 5432:5432 \
  postgres:16

# immediately verify BEFORE trusting it:
docker inspect my-postgres --format "{{.Mounts}}"
# → exactly one mount, correct path
New rule going forward: always verify a mount immediately with docker inspect before trusting it, and always get exact internal data paths from the image's official Docker Hub documentation — never type them from memory.

Common data paths (from official images — never guess these)

postgres   →  /var/lib/postgresql/data
mysql      →  /var/lib/mysql
mongodb    →  /data/db
redis      →  /data

No consistent pattern across images — each application decides its own internal path. Volume mounting isn't "Docker decides where data goes" — it's "you tell Docker where to put persistent storage, and the application decides where it actually writes files. Your job is making those two locations match."

Backing up a named volume's data

docker run --rm -v pgdata:/data -v C:/backup:/backup ubuntu \
  tar czf /backup/pgdata-backup.tar.gz /data

A throwaway container mounts both the named volume and a host folder just to copy data between them — the database itself stays on the named volume permanently; you don't switch it to a bind mount just to peek at files.

one more concept, confirmed by reasoning Could two containers (my-postgres and my-postgres-2) use the same pgdata volume simultaneously? No — PostgreSQL locks its data directory on startup. Only one process should write to those files at once, or you'd risk data corruption from two independent instances writing conflicting changes to the same files. Real production systems solve "I need multiple instances" with database replication, or by pointing multiple app services at one shared database container over the network — which leads directly into Chapter 6.
