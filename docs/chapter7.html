Docker Compose

Why Compose exists

Getting Flask and Postgres talking manually (Chapter 6) took four separate commands:

docker network create my-app-network
docker volume create pgdata
docker run -d --name my-postgres --network my-app-network \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16
docker run -d --name flask-app --network my-app-network \
  -p 5000:5000 \
  flask-app:v2

Each with multiple flags, run in a specific order — get the order wrong, forget a flag, or typo a path (as happened twice already), and you're debugging for an hour. A real production stack might have 5 services, each with volumes, networks, and env vars. Compose describes the entire stack in one YAML file, brought up with a single command.

analogy Individual docker run commands are verbal instructions for assembling furniture piece by piece. A Compose file is the IKEA manual — one document, fully specified, reproducible by anyone who opens it.

Anatomy of a Compose file

services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data

  flask-app:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - postgres

volumes:
  pgdata:

Every container is a service — the YAML key (postgres, flask-app) is the service name. build: . instead of image: tells Compose to run docker build automatically from the Dockerfile in the current directory. depends_on controls startup order.

Compose's automatic networking

Notably absent from the file: docker network create. When you run docker compose up, Compose automatically: 

1. Creates ONE user-defined network for this entire project
   (named something like "chapter7_default")
2. Attaches EVERY service in the file to that network
3. Each service becomes reachable by its SERVICE NAME
   (not container_name — the YAML key itself)

Exactly what was done manually in Chapter 6 — Compose just does it silently, every time.

service name vs. container_name — not the same thing DNS resolution on the Compose network uses the service name (the YAML key), not container_name, which is purely cosmetic — it only affects what shows in docker ps/docker logs and has zero effect on networking. Removing container_name entirely changes nothing about whether other services can reach it — the service name always works as the hostname, with or without it.

Running it, and reading the evidence

docker compose up

docker compose ps
# NAME                   SERVICE     STATUS
# chapter7-flask-app-1   flask-app   Up
# chapter7-postgres-1    postgres    Up

docker network ls
# chapter7_default   bridge   local   ← Compose's auto-created network

docker volume ls
# chapter7_pgdata                     ← Compose's auto-created volume

Naming pattern: <project-folder>-<service-name>-<instance-number>. The -1 exists because Compose supports scaling multiple instances of the same service.

a fresh project means a fresh volume chapter7_pgdata is a completely different, brand-new volume from Chapter 6's pgdata — proven by checking docker volume ls directly. Predicting correctly ahead of time: a fresh volume means /students would fail with the same relation "students" does not exist error, since nothing has ever written to this volume yet — confirmed on the first request.

Fixing it properly — automatic DB initialization

Rather than manually running psql every time, the official postgres image auto-executes any .sql file placed in /docker-entrypoint-initdb.d/ — but only the very first time it starts against a genuinely empty data directory. 

-- init.sql
CREATE TABLE students (id SERIAL PRIMARY KEY, name TEXT);
INSERT INTO students (name) VALUES ('Anshul');
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql

  flask-app:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - postgres
    environment:
      DB_HOST: postgres

volumes:
  pgdata:

./init.sql:/docker-entrypoint-initdb.d/init.sql is a bind mount — mapping a local file into the exact folder Postgres checks on startup.

init only runs on a genuinely fresh volume Since chapter7_pgdata already had system files in it from the earlier failed attempt, simply adding init.sql and restarting wasn't enough — Postgres would still skip initialization, same as always. Fix: force a truly fresh volume. 
docker compose down -v   # -v also removes volumes declared in this file
docker compose up
docker compose down alone keeps volumes intact (safe, data survives). Adding -v explicitly destroys them — never run -v in a real project without being certain.

depends_on ≠ ready

the real gap depends_on only guarantees the dependency's container process starts before the dependent service starts — not that it's actually finished initializing and ready to accept connections. 
Timeline:
0s   postgres container process starts
0-3s postgres is initializing internally (system files, init.sql, opening its port)
1s   flask-app starts (depends_on satisfied — postgres STARTED)
1s   flask-app tries to connect → CONNECTION REFUSED (not listening yet!)
3s   postgres finally ready
A real, common production bug — a race condition that may not fail every time, which makes it worse than a consistent failure (harder to reproduce and debug).

The fix: health checks, which actually verify readiness rather than just process start:

services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  flask-app:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      DB_HOST: postgres

depends_on: postgres: condition: service_healthy means "don't start flask-app until Postgres's healthcheck actually passes" — not just "until the process started."

--build vs. plain up

why code changes can silently not apply docker compose up only triggers a build automatically the very first time (when no image exists yet). Every time after that, it reuses whatever image is cached, completely unaware the code changed. 
docker compose up --build
Rule: any time the Dockerfile, requirements.txt, or app code changes, use --build — or the change silently won't apply. Caught this directly: added Redis to app.py, ran plain docker compose up, and the /students response was missing the new "source" field entirely — proof the old, cached image was still running. docker compose down && docker compose up --build fixed it.

a separate concern: init.sql changes Changing init.sql needs down -v + up (force a fresh volume), not --build — since init scripts are bind-mounted in, not baked into the image at all. Code changes and data/schema changes are two independent problems with two independent fixes.

Mini Project 4 — adding Redis for caching

Redis is an in-memory key-value store — extremely fast, typically used for temporary or frequently-accessed data rather than permanent records. Real pattern: cache /students so repeated requests don't all hit Postgres. 

Without caching:                  With Redis caching:
Request → Flask → Postgres        Request 1 → Flask → Postgres → Redis (store)
Request → Flask → Postgres        Request 2 → Flask → Redis (fast, no Postgres hit)
Request → Flask → Postgres        Request 3 → Flask → Redis (fast, no Postgres hit)
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

  flask-app:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      DB_HOST: postgres
      REDIS_HOST: redis

volumes:
  pgdata:
import redis, json

redis_client = redis.Redis(
    host=os.environ.get('REDIS_HOST', 'redis'),
    port=6379,
    decode_responses=True
)

@app.route('/students')
def get_students():
    cached = redis_client.get('students_cache')
    if cached:
        return {"students": json.loads(cached), "source": "cache"}

    conn = psycopg2.connect(
        host=os.environ.get('DB_HOST', 'postgres'),
        database='postgres', user='postgres',
        password=os.environ.get('DB_PASSWORD', 'mysecretpassword')
    )
    cur = conn.cursor()
    cur.execute('SELECT id, name FROM students;')
    rows = cur.fetchall()
    cur.close(); conn.close()

    students = [{"id": r[0], "name": r[1]} for r in rows]
    redis_client.setex('students_cache', 30, json.dumps(students))
    return {"students": students, "source": "database"}

Check Redis first. On a hit, return immediately — never touches Postgres. On a miss, query Postgres, then setex ("set with expiry") stores the result for 30 seconds. After that, the next request naturally falls through to Postgres again, keeping the cache from serving stale data forever — a real tradeoff every caching system makes between performance and freshness.

full cache lifecycle, predicted correctly then verified 
Request 1 (fresh restart): source = "database"   (nothing cached yet)
Request 2, 3 (within 30s): source = "cache"       (Redis hit, no Postgres touch)
Request 4 (after 35s+):    source = "database"    (key expired, re-cached)
Exactly matches how setex('students_cache', 30, ...) is designed to behave — verified live by hitting the endpoint repeatedly and watching the source field flip at precisely the expected boundary.

Project-scoped naming — why it matters

Compose prefixes volumes and networks with the project folder name (chapter7_pgdata, chapter7_default). Without this, two unrelated projects on the same machine both naming a volume pgdata would collide — Project B's database could accidentally read or corrupt Project A's data just from a name clash. Prefixing guarantees every project gets isolated volumes and networks by default, even when the same short names are reused inside each YAML file.

Compose command reference

docker compose up              # start (foreground)
docker compose up -d           # start (detached)
docker compose up --build      # rebuild images first — use after ANY code change
docker compose ps              # project-scoped container list
docker compose logs -f <svc>   # follow logs for one service
docker compose down            # stop + remove containers/network, KEEP volumes
docker compose down -v         # also DELETE volumes
docker compose restart <svc>   # restart one service
