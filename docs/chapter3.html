Images: The Blueprint

Why "Pulling from library/nginx" shows multiple lines

Pulling an image always prints several lines like Pull complete, not just one. That's because an image is not a single file — it's a stack of layers, and each one gets pulled separately. 

Pulling from library/nginx
2cc3ae149d28: Pull complete
1018f2b8dba8: Pull complete
b831e78d8e20: Pull complete

The layered filesystem — the core insight

Each layer represents a set of filesystem changes — files added, modified, or deleted. Every layer is read-only and immutable — once created, it never changes. 

┌─────────────────────────────────┐
│         nginx Image             │
│  Layer 4: nginx config files    │  ← added last
│  Layer 3: nginx binary          │
│  Layer 2: system libraries      │
│  Layer 1: base Ubuntu/Debian    │  ← added first
└─────────────────────────────────┘

When Docker runs a container from an image, it adds one more layer on top — a thin writable layer, unique to that container: 

┌─────────────────────────────────┐
│  Container Writable Layer   R/W │  ← your container's changes
├─────────────────────────────────┤
│  Layer 4: nginx config      R/O │  ← image layers
│  Layer 3: nginx binary      R/O │     (read only)
│  Layer 2: system libraries  R/O │
│  Layer 1: base OS           R/O │
└─────────────────────────────────┘

Union filesystem — the magic that makes it work

A union filesystem merges multiple layers into a single unified view. The container sees one clean filesystem — not a visible stack — even though underneath, those files are spread across several read-only layers plus one writable layer on top.

analogy Imagine four transparent glass sheets stacked on top of each other. Sheet 1 has a sky painted on it, sheet 2 has mountains, sheet 3 has trees, sheet 4 has a house. Looking through all of them together, you see one complete painting — but each sheet remains independent and unchanged.

Copy-on-write (CoW) — how containers modify files

Containers are supposed to be isolated — but what happens when a container needs to modify a file that lives in a read-only image layer?

the mechanism When a container wants to modify a file from a read-only layer, Docker first copies that file up to the writable layer, then modifies the copy. The original in the image layer is never touched. 
Container wants to edit /etc/nginx/nginx.conf

Step 1: File currently lives in Layer 4 (read-only)

Step 2: Docker copies it up to writable layer
        ┌─────────────────────────────────┐
        │  Writable Layer                 │
        │  /etc/nginx/nginx.conf (copy) ← │ modified here
        ├─────────────────────────────────┤
        │  Layer 4: nginx.conf (original) │ untouched
        └─────────────────────────────────┘

Step 3: Container sees the modified copy (top layer wins)
Step 4: Other containers using same image still see original
verified live Ran two separate ubuntu containers from the same image. Wrote /tmp/myfile.txt inside the first one. Checked for it inside the second — it wasn't there. The file only ever existed in the first container's own writable layer: 
cow-test container:                cow-test-2 container:
┌─────────────────────┐            ┌─────────────────────┐
│ Writable Layer      │            │ Writable Layer      │
│ /tmp/myfile.txt ✅  │            │ (empty)             │
├─────────────────────┤            ├─────────────────────┤
│ Ubuntu image layer  │            │ Ubuntu image layer  │
│ (shared, read-only) │            │ (shared, read-only) │
└─────────────────────┘            └─────────────────────┘
This is exactly why five containers from the same image never interfere with each other, and why they don't each need their own full copy of the image.

Why layers matter — the caching superpower

Imagine two apps that both start from Ubuntu, install the same system libraries, and both run Python 3.11: 

App A image:                App B image:
┌────────────────┐          ┌────────────────┐
│ App A code     │          │ App B code     │
│ Python 3.11    │          │ Python 3.11    │
│ system libs    │          │ system libs    │
│ Ubuntu 22.04   │          │ Ubuntu 22.04   │
└────────────────┘          └────────────────┘

Without layer sharing, that duplicated data gets stored twice. With layers, Docker recognizes identical layers and stores each one only once, shared across every image that uses it: 

Disk storage:
  Ubuntu 22.04 layer      ← stored ONCE, shared by both
  system libs layer        ← stored ONCE, shared by both
  Python 3.11 layer        ← stored ONCE, shared by both
  App A code layer         ← unique to App A
  App B code layer         ← unique to App B

This is exactly why pulling a second Python image is faster than the first — many layers are already local.

Image tags — versions and pointers, not copies

nginx:latest        → most recent stable build
nginx:1.25          → specific version
nginx:1.25-alpine   → version 1.25 on Alpine Linux (tiny)
python:3.11         → Python 3.11, full Debian image (~900MB)
python:3.11-slim    → Python 3.11, minimal Debian (~150MB)
python:3.11-alpine  → Python 3.11, Alpine Linux (~50MB)
the :latest trap :latest does not mean "automatically updates." It just means "whatever was tagged latest at the moment you pulled." Pull today, get Python 3.12. Pull again in 6 months, get 3.13 — with zero code changes on your end, your app can quietly break. In production, always pin exact versions: python:3.11.4-slim, never python:latest.

Tagging and pushing your own image

docker login

docker tag hello-world:latest anshul/my-first-image:v1.0

docker images
# REPOSITORY                    TAG     IMAGE ID
# hello-world                   latest  96498ffd522e
# anshul/my-first-image         v1.0    96498ffd522e  ← same ID!

docker push anshul/my-first-image:v1.0

docker rmi anshul/my-first-image:v1.0    # delete local copy
docker pull anshul/my-first-image:v1.0   # pull it back from the internet
what this proves Same Image ID, two different tag names — tagging never copies data, it just creates a new label pointing at the same underlying image, like a shortcut. This full round-trip — build locally, push to a registry, pull elsewhere — is exactly how real deployment works: CI/CD builds and pushes, your server pulls.

Housekeeping — docker system commands

docker system df           # how much disk Docker is using
docker system prune        # remove unused containers/images/networks
docker system prune -a     # more aggressive — also unused images
docker container prune     # only stopped containers
docker image prune         # only unused images
when NOT to run prune -a It's not always safe to run anytime. If you have images you haven't used recently but still need, prune -a deletes them — you'd have to re-pull or rebuild. Fine on a personal dev machine, risky on a shared or CI server.

Base image sizes — measured, not assumed

ImageSize
python:3.111.6 GB
python:3.11-slim188 MB
python:3.11-alpine84 MB

A 19x difference between full and Alpine. At scale that compounds fast:

100 deployments/day × 1.6GB = 160GB bandwidth daily
100 deployments/day × 84MB  =   8.4GB bandwidth daily

Alpine — why you'll see it everywhere

Base OS size comparison:
Ubuntu 22.04    ~80MB
Debian          ~50MB
Alpine          ~5MB

Alpine is dramatically smaller — faster downloads, less disk, smaller attack surface, faster CI/CD. The trade-off: Alpine uses musl libc instead of glibc, so some Python packages with C extensions don't compile cleanly without extra steps. slim (minimal Debian) is usually the safer middle ground for Python — most of the size win, none of the compatibility risk. 

For most Python apps:    python:3.11-slim   ✅
For Node.js apps:        node:18-alpine     ✅
For simple tools:        alpine             ✅
For maximum compat:      python:3.11        (only if genuinely needed)
