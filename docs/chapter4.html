Dockerfiles: Building Your Own Images

The big shift — from consumer to producer

Up to this point, every image used (nginx, ubuntu, hello-world) was built by someone else. From here, that changes — every real Docker project starts with writing a Dockerfile.

analogy A Dockerfile is a cooking recipe. It lists exact ingredients and exact steps in exact order. Anyone who follows the same recipe gets the same dish — on any machine, anywhere in the world. 
Recipe (Dockerfile)          Result (Image)
───────────────────          ──────────────
Start with Ubuntu      →     Base layer
Install Python         →     + Python layer
Copy my code           →     + Code layer
Install dependencies   →     + Dependencies layer
Set startup command    →     Final image ready to run

Docker reads the file top to bottom, executes each instruction, and each instruction creates one layer.

The absolute minimum Dockerfile

FROM ubuntu:22.04
CMD ["echo", "Hello from my first image!"]

FROM sets the base image — every Dockerfile must start with it. ubuntu:22.04 instead of ubuntu:latest, for the same reproducibility reason covered in Chapter 3.

CMD defines the default command that runs when a container starts — it does not run during docker build, only at docker run. The exec form ["echo", "..."] is preferred over shell form (CMD echo ...) — more predictable, handles signals correctly. 

docker build -t my-first-image:v1 .
docker run my-first-image:v1
# → Hello from my first image!

The trailing . in docker build is the build context — it tells Docker "send everything in this directory to the Daemon for the build." Large folders slow builds down even for files you don't actually need — solved with .dockerignore, which works exactly like .gitignore.

WORKDIR

Sets the working directory inside the container for all following instructions. If it doesn't exist, Docker creates it. Without it, files land in / — messy and potentially risky. Equivalent to mkdir /app && cd /app.

COPY

COPY <source on your machine>  <destination in image>
COPY requirements.txt .        → copy to current WORKDIR (/app)
COPY . .                       → copy everything to /app
COPY src/ /app/src/            → copy folder to specific path

RUN vs. CMD — the most important early distinction

RUN   → executes during IMAGE BUILD
        result is baked into the image layer, permanent

CMD   → executes when CONTAINER STARTS
        runs every time you do docker run
        not baked in — just metadata

EXPOSE

Documents that a container listens on a given port. It does not actually publish the port — that's what -p at docker run does. Think of EXPOSE as the Dockerfile telling other developers "this app uses port 5000," nothing more.

Layer caching — order matters, with real numbers

COPY requirements.txt .              # changes rarely
RUN pip install -r requirements.txt  # slow step — want this cached
COPY . .                             # changes constantly — put last
First build:
  Layer 1: FROM python:3.11-slim     → downloaded
  Layer 2: WORKDIR /app              → created
  Layer 3: COPY requirements.txt     → copied
  Layer 4: RUN pip install           → installed (slow, 30 seconds)
  Layer 5: COPY . .                  → copied

Change one line in app.py, rebuild:
  Layer 1: FROM python:3.11-slim     → CACHED ✅
  Layer 2: WORKDIR /app              → CACHED ✅
  Layer 3: COPY requirements.txt     → CACHED ✅ (unchanged)
  Layer 4: RUN pip install           → CACHED ✅ (requirements unchanged!)
  Layer 5: COPY . .                  → REBUILT (app.py changed)

pip install (the slow step) stays cached — rebuild takes 2 seconds instead of 35.

what happens if you get the order wrong 
COPY . .                              # includes app.py
RUN pip install -r requirements.txt
Now changing any line in app.py invalidates the COPY . . layer — which invalidates RUN pip install too, since every layer after a changed one gets rebuilt. 35 seconds wasted on every single code change, forever. Rule: most stable instructions at the top, most volatile (your code) at the bottom.

ENTRYPOINT vs. CMD — the distinction that trips everyone

# CMD only — fully replaceable
CMD ["python", "app.py"]
docker run flask-app:v1                    # runs: python app.py
docker run flask-app:v1 python other.py   # runs: python other.py (overridden)
docker run flask-app:v1 bash              # runs: bash (overridden)
# ENTRYPOINT only — runtime args are APPENDED, not replaced
ENTRYPOINT ["python", "app.py"]
docker run flask-app:v1                   # runs: python app.py
docker run flask-app:v1 --debug           # runs: python app.py --debug
docker run flask-app:v1 bash             # runs: python app.py bash (appended!)
# Using both together — the most flexible pattern
ENTRYPOINT ["python"]
CMD ["app.py"]
docker run flask-app:v1              # runs: python app.py
docker run flask-app:v1 other.py    # runs: python other.py (CMD overridden)
docker run flask-app:v1 --version   # runs: python --version
mental model ENTRYPOINT = the verb (always "python"). CMD = the noun (default "app.py", but changeable). Web servers typically use CMD only (simple, flexible). CLI tools often use both. Database images tend to fix behavior with ENTRYPOINT alone.

ADD vs. COPY

ADD does everything COPY does, plus auto-extracting tar archives and downloading from URLs. Best practice: always use COPY unless you specifically need those extras — COPY is explicit and predictable; ADD's automatic behavior can surprise you.

ENV — runtime-configurable defaults

ENV APP_ENV=production
ENV PORT=5000
import os
port = int(os.environ.get('PORT', 5000))
app_env = os.environ.get('APP_ENV', 'development')

ENV in a Dockerfile is a default, not a locked value — -e at docker run always overrides it: 

docker run -e DB_HOST=postgres-server -e APP_ENV=staging flask-app:v1
why this matters This is the whole point of ENV: build once, configure at runtime. Same image ID deployed to staging, testing, and production — only the environment variable changes at the door. The exact image you tested is the exact image you deploy; nothing gets rebuilt between environments, so there's no "worked in staging, broke in prod because something was different" risk.

ARG — build-time-only variables

ARG PYTHON_VERSION=3.11
FROM python:${PYTHON_VERSION}-slim

ARG APP_VERSION=1.0.0
RUN echo "Building version ${APP_VERSION}"
# docker build --build-arg APP_VERSION=2.1.0 -t myapp:v2 .

ARG → build config. ENV → application config, available at runtime too.

USER — don't run as root

By default, everything in a container runs as root. If an attacker exploits the app and breaks out of the container, they're root inside it — a real security risk. 

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .

RUN useradd --create-home appuser && \
    chown -R appuser:appuser /app
USER appuser

EXPOSE 5000
CMD ["python", "app.py"]
real bug: the ownership trap COPY runs as root by default — before USER appuser takes effect — so every copied file is owned by root:root. Switching to a non-root user afterward without a chown -R means that user technically can't write to its own app directory. This often looks fine in a simple demo (just reading one Python file), but breaks the moment the app needs to write logs or handle uploads. Verified the fix directly: 
appuser@36a6668d33a8:/app$ whoami
appuser
appuser@36a6668d33a8:/app$ ls -la /app
drwxr-xr-x 1 appuser appuser  .idea
-rwxr-xr-x 1 appuser appuser  app.py
-rwxr-xr-x 1 appuser appuser  requirements.txt
Every file owned by appuser, not root — proof the chown -R appuser:appuser /app line actually worked.

Small habit worth building: after every build, glance inside the running container (ls -la /app) and ask "does anything here look like it shouldn't ship?" — IDE folders, .env secrets, and test files sneak into images constantly if this file isn't maintained.

0.0.0.0 vs. 127.0.0.1

Flask defaults to binding on 127.0.0.1 — accessible only from inside the container itself. Using host='0.0.0.0' means "listen on all network interfaces," which is what makes the app reachable from outside the container through the -p port mapping. Always use this in Docker.

Complete professional Dockerfile

# Pin exact version - reproducibility
FROM python:3.11.4-slim

LABEL maintainer="anshul@example.com"
LABEL version="1.0"

ARG APP_ENV=production

ENV APP_ENV=${APP_ENV} \
    PORT=5000 \
    PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

RUN useradd --create-home appuser && \
    chown -R appuser:appuser /app
USER appuser

EXPOSE 5000

CMD ["python", "app.py"]

PYTHONDONTWRITEBYTECODE=1 skips creating .pyc files (cleaner image). PYTHONUNBUFFERED=1 prints logs immediately instead of buffering them — critical for actually seeing logs in real time via docker logs.

Mini Project 1 — a real Flask API, built and debugged

Three endpoints, proving ENV values genuinely flow from Dockerfile → container → Python code:

from flask import Flask
import sys
import os

app = Flask(__name__)

@app.route('/')
def home():
    return "Docker Learning API - Chapter 4"

@app.route('/health')
def health():
    return {"status": "healthy", "version": "1.0"}

@app.route('/info')
def info():
    return {
        "python_version": sys.version,
        "environment": os.environ.get('APP_ENV', 'development'),
        "port": os.environ.get('PORT', '5000')
    }

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000, debug=False)
a Dockerfile bug caught by dry-run, before ever building A multi-line ENV continuation using a trailing backslash accidentally started a new line by itself (\PORT=5000) — invalid syntax. Several instructions (RUN, COPY, USER, EXPOSE, CMD) also ended up squashed onto a single line. A Dockerfile executes one instruction per line — multiple instructions on one line get misread as arguments to the first one. Fixed by giving each instruction its own line, and using proper line continuation: 
ENV APP_ENV=${APP_ENV} \
    PORT=5000

Final verified output, confirming the full ENV override chain works end-to-end:

docker build -t docker-api:v1 .
docker run -d -p 5000:5000 -e APP_ENV=staging --name api-test docker-api:v1

curl http://localhost:5000/
# Docker Learning API - Chapter 4

curl http://localhost:5000/health
# {"status":"healthy","version":"1.0"}

curl http://localhost:5000/info
# {
#   "environment": "staging",
#   "port": "5000",
#   "python_version": "3.11.4 (main, Aug 16 2023, ...) [GCC 12.2.0]"
# }

"environment": "staging" confirms the -e APP_ENV=staging override beat the Dockerfile's development default — runtime always wins over build-time. "port": "5000" confirms the Dockerfile's ENV PORT=5000 default was read correctly. The Python version string is live proof this is genuinely running inside the container's own Python 3.11.4, not some Python on the host machine.
