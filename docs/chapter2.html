Docker Architecture & Core Concepts

Who's actually doing the work?

Every time you type docker run nginx, something happens behind the scenes involving three separate components talking to each other — not one monolithic "Docker" doing everything. 

┌─────────────────────────────────────────────────────┐
│   YOU                                               │
│    │                                                │
│    ▼                                                │
│  Docker CLI          ← the thing you type into      │
│    │                                                │
│    │  REST API calls (HTTP)                         │
│    ▼                                                │
│  Docker Daemon       ← the brain, always running    │
│  (dockerd)                                          │
│    │                                                │
│    ├──── manages ────► Containers                   │
│    ├──── manages ────► Images                       │
│    ├──── manages ────► Volumes                      │
│    ├──── manages ────► Networks                     │
│    │                                                │
│    │  pulls from                                    │
│    ▼                                                │
│  Docker Registry     ← where images live remotely   │
│  (Docker Hub)                                       │
└─────────────────────────────────────────────────────┘

Component 1 — Docker CLI

The only part you directly interact with. It does zero actual work — it just translates your command into an API call and sends it to the Daemon. Think of it as a waiter: it carries your order to the kitchen, but never cooks anything itself.

Component 2 — Docker Daemon (dockerd)

The actual brain, running silently in the background at all times. It creates and runs containers, builds images, manages volumes and networks, and talks to Docker Hub to pull images. This is the kitchen — it receives orders and does the real cooking.

Component 3 — Docker Registry

Where images are stored and shared remotely. Docker Hub is the default public registry — like a supermarket the kitchen visits when it needs an ingredient (image) it doesn't already have locally.

Tracing docker run -d -p 8080:80 nginx, step by step

Step 1:  You type → docker run -d -p 8080:80 nginx

Step 2:  CLI parses the command
         Builds an API request:
         POST /containers/create
         {image: "nginx", ports: "8080:80", detach: true}

Step 3:  CLI sends this to the Daemon via REST API

Step 4:  Daemon receives it
         Checks: "Do I have nginx image locally?"
         Answer: NO (first time)

Step 5:  Daemon contacts Docker Hub
         GET hub.docker.com/nginx:latest
         Downloads the image layer by layer

Step 6:  Image stored locally in Daemon's image cache

Step 7:  Daemon creates a container from the image
         Assigns it: a unique ID, a name, a network interface,
         an isolated filesystem, port mapping (8080 → 80)

Step 8:  Daemon starts the container process
         nginx web server begins running inside

Step 9:  CLI receives confirmation, prints container ID,
         returns control to you (because of -d flag)

Why the second run is instant

Run the same image again and there's no "Pulling from library/nginx" message — it starts almost immediately. That's Step 4 shortcutting: the Daemon checks locally first, finds the image already cached, and skips straight to Step 7. 

First run:   CLI → Daemon → Hub (download) → Create → Start
Second run:  CLI → Daemon → Local cache → Create → Start

The Docker object model

┌─────────────────────────────────────────────────┐
│              Docker Objects                     │
│  📦 Images      → blueprints (read-only)        │
│  🏃 Containers  → running instances of images   │
│  💾 Volumes     → persistent storage            │
│  🌐 Networks    → communication between         │
│                   containers                    │
└─────────────────────────────────────────────────┘
stop and think If you run 5 containers from the same nginx image simultaneously, how many copies of the image exist on disk? Still just one. The image is read-only; all five containers read from the same shared layers. Each container only stores its own changes on top — this is exactly what makes containers lightweight, and it's covered in full in Chapter 3.

docker run is really docker create + docker start combined — that's why it's the command you reach for most.

stopped ≠ deleted A stopped container is like a switched-off laptop — docker start brings it back exactly as it was. A removed container (docker rm) is like a laptop that's been thrown away — you'd need to docker run a completely fresh one.

Essential commands, with real understanding (not memorization)

docker ps                    # running containers only
docker ps -a                 # ALL containers (running + stopped)
docker start <name/id>       # start a stopped container (reuse existing)
docker stop <name/id>        # stop gracefully
docker kill <name/id>        # force kill immediately
docker rm <name/id>          # remove a stopped container
docker rm -f <name/id>       # remove a running container forcefully
docker images                # list images
docker rmi <image>           # remove an image
docker pull ubuntu           # pull without running
docker logs <name/id>        # view container logs
docker exec -it <name/id> bash   # enter a running container's shell

How isolation actually works

Two Linux kernel features make container isolation possible.

Mechanism 1 — Namespaces (the illusion of isolation)

A namespace wraps a global system resource and makes a process think it has its own private version of it. Docker uses six per container: 

┌─────────────────────────────────────────────────────┐
│              Container's Private View               │
│  PID namespace    → its own process list             │
│  NET namespace    → its own network interface + IP   │
│  MNT namespace    → its own filesystem view          │
│  UTS namespace    → its own hostname                 │
│  IPC namespace    → its own inter-process comms      │
│  USER namespace   → its own user IDs                 │
└─────────────────────────────────────────────────────┘
analogy Imagine a hotel where every floor has its own "Room 101" on the map. Guests don't share rooms — they each have a private view of the same building. Namespaces work the same way: your app inside a container thinks it's Process 1, even though a "real" Process 1 also exists on the host. Two different namespaces, no conflict, no visibility into each other.

Mechanism 2 — Cgroups (the resource police)

Namespaces handle isolation. Cgroups (control groups) handle resource limits — how much CPU, RAM, disk I/O, and network a process can actually consume. 

Without cgroups:
Container A goes rogue, uses 100% CPU
→ Container B starves → whole machine slows down
→ production is down → you're getting paged at 3am

With cgroups:
Container A limited to 25% CPU, 512MB RAM
→ even if it goes rogue, it can't take more
→ Container B unaffected → you sleep peacefully
docker run -d --memory="512m" --cpus="0.5" nginx
putting it together A container = a process + namespaces (isolated view) + cgroups (limited resources) + a filesystem from the image. No mini-OS, no magic — just a process with walls around it and a ceiling above it. That's why containers start in milliseconds: there's no OS to boot, just a process starting with kernel features wrapping it in isolation.

docker exec — entering a running container

docker run -d -p 8080:80 --name my-nginx nginx
docker exec -it my-nginx bash
# prompt becomes: root@0ab4cf6d5bc4:/#
ls
cat /etc/os-release
exit

-i keeps stdin open (interactive), -t allocates a terminal so you actually see a prompt. Use -it when exploring or debugging live. Use -d for background, long-running services like web servers and databases.

key rule, learned by breaking it A container lives only as long as its main process lives. In docker run -it ubuntu bash, bash is the main process. Typing exit stops bash — and since there's no other process keeping the container alive, the container stops too, immediately. This isn't a bug; it's the single most common cause of "why did my container exit immediately?" confusion, and it explains itself once you know this rule.

Verified directly: running docker exec -it my-nginx bash after docker rm my-nginx returns Error: No such container — because removal (not stopping) permanently deletes the container, so there's nothing left to exec into.

Docker Hub — the registry, in more detail

┌─────────────────────────────────────────────────┐
│                  Docker Hub                     │
│  Official Images        Community Images        │
│  ┌─────────────┐        ┌─────────────┐         │
│  │   nginx     │        │  user/app   │         │
│  │   postgres  │        │  company/api│         │
│  │   python    │        │  team/model │         │
│  │   ubuntu    │        └─────────────┘         │
│  │   redis     │                                │
│  └─────────────┘                                │
│  Maintained by          Maintained by           │
│  Docker/vendors         anyone                  │
└─────────────────────────────────────────────────┘

Official images (nginx, postgres, python) are maintained by Docker or the software vendors themselves — secure, optimized, regularly updated. Always prefer these over random community images when one exists for what you need. 

nginx                    → official image, latest tag
nginx:1.25               → official image, specific version
username/myapp           → community image
username/myapp:v2.1      → community image, specific version
