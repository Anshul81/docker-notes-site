Real-World Dockerization Patterns

The gap between "build" and "run"

Every Dockerfile so far has been single-stage — one FROM, build everything, ship it as-is. That works, but it means shipping build-time tools into production even when the running app never needs them again.

the core question Once an image is built, does the container still need pip itself — or the build tools/caches pip created along the way — or does it only need the installed packages and the app code? A carpenter's workshop full of saws and glue isn't shipped inside the finished chair.

Where the waste is obvious — a Node/React example

FROM node:18
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
RUN npm run build
CMD ["npm", "start"]

npm install pulls in every dev dependency — build tools, compilers, testing libraries. npm run build then produces a handful of static HTML/CSS/JS files. The final running container doesn't need Node.js, npm, or any dev dependency at all to serve static files. 

node:18 base image alone:     ~1 GB
+ node_modules (dev deps):    +500 MB - 1 GB
+ your actual built output:   maybe 5-10 MB
Final image: potentially 1.5-2 GB
Actually needed at runtime:   ~10 MB of static files

Multi-stage builds — the fix

A multi-stage Dockerfile has more than one FROM. Earlier stages do the heavy lifting; the final stage only copies over finished artifacts, discarding everything else. 

# Stage 1: "builder" — has all the heavy tools
FROM node:18 AS builder
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
RUN npm run build

# Stage 2: final image — lean, production-only
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80

AS builder names the first stage. The second FROM starts a completely separate image — none of Node's layers, none of node_modules, carry over automatically. COPY --from=builder /app/dist ... reaches back into the named stage and copies out only the compiled output. Node, npm, and every dev dependency stay trapped in the discarded builder stage — they never become part of the final image's layers at all. 

Single-stage:  Final image: ~1.5-2 GB (Node runtime + all dev deps + build output)
Multi-stage:   Final image: ~25-40 MB (just nginx:alpine + your static files)

Often a 40-50x size reduction — faster pulls, faster deploys, smaller attack surface.

Does this help every project equally? No.

ProjectBuild tool ≠ runtime need?Multi-stage benefit
Node/React static siteYes — Node vs. plain HTML/CSS/JSHuge (40-50x smaller)
Flask API (this course)Barely — packages needed at runtime tooSmall, only if a package needs a compiler
Static docs site (nginx:alpine)No build step exists at allNone — nothing to separate into stages
the real rule Multi-stage builds help exactly when a project's build-time footprint is meaningfully bigger than its run-time footprint. No such gap, no real benefit — just added complexity for nothing. Python doesn't "compile away" its dependencies the way a frontend build does — from flask import Flask needs the real flask package present every time the app starts, so there's no separate "build artifact" distinct from "the thing that runs."

Where multi-stage still helps Python: discarding compilers only needed to install certain packages.

FROM python:3.11.4-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

FROM python:3.11.4-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "app.py"]

Dev vs. production Compose configs

A bind mount for live editing makes no sense on a production server — there's no local source folder being actively edited there, and tying production behavior to whatever happens to sit in a server folder defeats the point of a self-contained image.

Development wantsProduction wants
Bind mount (live edit)Baked-in code (COPY . . in image)
Flask dev server (debug=True)Real WSGI server, debug off
Exposed DB port (debugging)DB port NOT exposed publicly

Compose file layering

docker-compose.yml           ← shared base: services, images, networks
docker-compose.override.yml  ← dev-only additions (auto-applied by default!)
docker-compose.prod.yml      ← prod-only additions (applied explicitly)
# docker-compose.yml (base — identical in every environment)
services:
  flask-app:
    build: .
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      DB_HOST: postgres

  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pgdata:
# docker-compose.override.yml (dev — auto-merged, zero extra flags)
services:
  flask-app:
    volumes:
      - .:/app
    ports:
      - "5000:5000"
    environment:
      FLASK_DEBUG: "1"
  postgres:
    ports:
      - "5432:5432"
# docker-compose.prod.yml (prod — must be specified explicitly)
services:
  flask-app:
    environment:
      FLASK_DEBUG: "0"
    restart: unless-stopped
  postgres:
    restart: unless-stopped
docker compose up
# → dev: override.yml merges in automatically, zero extra flags needed

docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
# → prod: explicit -f flags required
how merging actually works Overriding one setting doesn't wipe out the rest of that section — Compose merges keys within a block. Base file's DB_HOST and prod override's FLASK_DEBUG both survive into the final config. Verify any merge before trusting it: docker compose -f docker-compose.yml -f docker-compose.prod.yml config prints the complete merged YAML with nothing actually started.
why this must be structural, not memory-based If docker-compose.override.yml (dev settings — bind mount, exposed DB port, debug mode) ever ends up sitting on a production server, plain docker compose up merges it in automatically, silently — a production database's port could end up publicly exposed, or the app could run in debug mode (leaking stack traces to anyone who triggers an error), without anyone explicitly asking for either. Relying on "always remember to type -f flags in prod" is a human-memory dependency — exactly the kind of failure point Docker's whole design philosophy tries to eliminate (same lesson as the manual READMEs from Chapter 1). The safer fix: never deploy the dev override file to the production server at all, so there's structurally nothing for a plain up to merge in.

Health checks — beyond "is the process alive"

docker ps's STATUS column only reflects whether a container's main process is still alive — not whether the application inside is actually responding correctly. A Flask worker can hang forever waiting on a dead database connection while still showing Up 2 hours. 

healthcheck:
  test: ["CMD-SHELL", "pg_isready -U postgres"]
  interval: 5s
  timeout: 5s
  retries: 5
test:      the actual command run to check health
interval:  how often to run it
timeout:   how long to wait before considering it failed
retries:   how many consecutive failures before UNHEALTHY
docker ps
# STATUS: Up 2 hours (healthy)
# STATUS: Up 2 hours (unhealthy)
# STATUS: Up 2 hours (health: starting)

Writing one for Flask itself

flask-app:
  build: .
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:5000/health"]
    interval: 10s
    timeout: 3s
    retries: 3
    start_period: 5s

This curl runs inside the Flask container itself — so localhost:5000 correctly refers to Flask running in that same container (which is bound to 0.0.0.0, accepting connections including from its own localhost). start_period gives a grace window before failed checks count against retries, so a container isn't marked unhealthy just for still being in the middle of starting up.

real gotcha: curl doesn't exist in slim images python:3.11-slim doesn't ship with curl by default — minimal images strip out anything not essential. A healthcheck calling curl without installing it first fails immediately, every time, marking the container unhealthy even though Flask itself runs fine. 
FROM python:3.11.4-slim

RUN apt-get update && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
...
This install belongs near the very top of the Dockerfile — even above COPY requirements.txt . — since it's tied to the base OS, not the app's dependencies or code. Editing app.py a hundred times over the project's life will never invalidate this layer. rm -rf /var/lib/apt/lists/* cleans up the package index cache afterward, since it's only needed during the install itself.
