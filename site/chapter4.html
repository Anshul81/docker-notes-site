Dockerfiles: Building Your Own Images

A Dockerfile is a recipe — a plain text file that tells Docker how to build an image, layer by layer, top to bottom.

Core instructions

InstructionRunsPurpose
FROMbuildBase image — always first line
WORKDIRbuildSets working directory for everything after
COPYbuildCopies files from build context into the image
RUNbuildExecutes a command, result baked into a layer permanently
ENVbuild + runtimeSets a default env var, overridable at docker run
ARGbuild onlyBuild-time-only variable
EXPOSEdocumentationDocuments the port — doesn't actually publish it
USERbuild + runtimeSwitches to a non-root user
CMDruntimeDefault command — fully overridable at docker run

RUN vs. CMD

RUN executes during docker build and is permanently baked into the image. CMD executes every time a container starts — it's not baked in, just metadata about what to run.

ENTRYPOINT vs. CMD

# CMD only — fully replaceable
CMD ["python", "app.py"]
docker run myapp bash        # runs: bash (CMD overridden)

# ENTRYPOINT only — runtime args are APPENDED, not replaced
ENTRYPOINT ["python", "app.py"]
docker run myapp --debug     # runs: python app.py --debug

# Both together — most flexible pattern
ENTRYPOINT ["python"]
CMD ["app.py"]
docker run myapp other.py    # runs: python other.py

Mental model: ENTRYPOINT = the verb, CMD = the default noun (changeable).

Layer caching — order matters

Docker caches each layer. If a layer's input hasn't changed, the cached version is reused instead of rebuilt. Put things that change rarely at the top of the Dockerfile, and things that change constantly (your code) at the bottom. 

COPY requirements.txt .              # changes rarely
RUN pip install -r requirements.txt  # cached unless requirements.txt changes
COPY . .                             # changes constantly — put last
common mistake COPY . . before RUN pip install means every single code change invalidates the pip install layer too — wasting 30+ seconds on every rebuild for no reason.

0.0.0.0 vs. 127.0.0.1

Flask defaults to 127.0.0.1 — accessible only from inside the container. Use host='0.0.0.0' to accept connections from outside the container (i.e. through the -p port mapping).

ENV as runtime configuration

ENV in a Dockerfile sets a default, not a locked value — -e at docker run always overrides it. 

docker run -e APP_ENV=staging myapp

This is the whole point: build once, configure at runtime. One image, deployed identically everywhere, only its environment variables differ between staging/testing/production. No rebuild-per-environment risk.

Non-root users — and the ownership trap

COPY . .
RUN useradd --create-home appuser && \
    chown -R appuser:appuser /app
USER appuser
real bug hit COPY runs as root by default (before USER is set), so copied files are owned by root. Switching to a non-root user afterward without chown -R means that user can't write to its own app directory — fine for simple read-only demos, but breaks the moment the app needs to write logs or handle uploads.
