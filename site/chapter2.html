Docker Architecture & Core Concepts

Three components

Typing docker run nginx involves three separate pieces talking to each other:

  • CLI — the waiter. Translates your command into an API call. Does zero actual work.
  • Daemon (dockerd) — the kitchen. Always running in the background, does the actual work: builds images, runs containers, manages volumes/networks.
  • Registry (Docker Hub) — the supermarket. Where images are stored remotely.

Tracing docker run -d -p 8080:80 nginx

CLI parses command → sends REST API request to Daemon
Daemon checks: image local? → NO → pulls from Docker Hub
Image cached locally
Daemon creates container: ID, name, network, filesystem, port mapping
Daemon starts the container process
CLI prints container ID, returns control (because of -d)

Second run of the same image skips the pull entirely — the local image cache means docker run goes straight from "check locally" to "create and start."

Container lifecycle

docker create → CREATED
docker start  → RUNNING
docker stop   → STOPPED (exited)
docker start  → RUNNING again (reusable!)
docker rm     → DELETED (permanent)
important distinction Stopped ≠ Deleted. A stopped container is a switched-off laptop — docker start brings it back exactly as it was. A removed container is gone permanently; you have to docker run a fresh one.

How isolation actually works

Two Linux kernel features make containers possible:

Namespaces — give a process its own private view of a system resource: PID (process list), NET (network interface), MNT (filesystem), UTS (hostname), IPC, USER. Like every hotel floor having its own "Room 101" — no actual conflict, no visibility into each other.

Cgroups — limit how much CPU/RAM/disk a container can actually use, enforced by the kernel. Without limits, one runaway container can starve everything else on the host. 

docker run -d --memory="512m" --cpus="0.5" nginx
mental model A container = a process + namespaces (isolated view) + cgroups (limited resources) + a filesystem from the image. No mini-OS, no magic — just a walled, capped process. That's why containers start in milliseconds.

docker exec — entering a running container

docker exec -it my-nginx bash

-i keeps stdin open, -t allocates a terminal. Use -it when you want to interact live (debugging, exploring). Use -d for background services that should just keep running.

key rule A container lives only as long as its main process lives. If that process is bash and you type exit, bash stops — and since bash was the container's main process, the container stops too. This isn't a bug; it explains most "why did my container exit immediately" confusion later.
