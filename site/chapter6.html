Networking in Docker

-p is not container-to-container

-p 5000:5000 connects your laptop to a container — it has nothing to do with letting two containers reach each other. Two completely separate mechanisms, easy to conflate at first.

The default bridge network's limitation

Every container is attached to Docker's default bridge network unless told otherwise. On this default network, containers can only reach each other by IP address — name-based DNS resolution doesn't work there. Since internal IPs change on every restart, this makes the default bridge impractical for real multi-container apps.

User-defined networks fix this

docker network create my-app-network

docker run -d --name my-postgres --network my-app-network \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

docker run -d --name flask-app --network my-app-network \
  -p 5000:5000 \
  flask-app:v1

On a user-defined network, Docker provides working DNS — any container can reach another by its container name, and that resolution stays correct even after a restart changes the underlying IP.

verified two ways python3 -c "import socket; print(socket.gethostbyname('my-postgres'))" returned 172.18.0.2 — and docker network inspect my-app-network independently confirmed the exact same IP for that container. Two sources agreeing, not just a lucky result.

Real connection string

DB_HOST = os.environ.get('DB_HOST', 'my-postgres')  # container name, not IP

Same ENV-as-default pattern from Chapter 4, now applied to networking — the container name is stable and overridable, unlike a hardcoded IP.

Useful commands

docker network ls
docker network create my-app-network
docker network inspect my-app-network
