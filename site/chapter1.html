Why Docker Exists

The problem: "it works on my machine"

Before Docker, this was the norm. You build something on your laptop, hand it to a teammate, and it breaks — not because the code is wrong, but because the environment is different. Different Python version, different library versions, different OS, missing environment variables.

How people tried to solve it before Docker

Era 1 — the manual README. Step-by-step setup instructions. Fragile, slow, easy to get wrong.

Era 2 — Virtual Machines. Ship the entire computer, OS included. This worked, but each VM carries a full OS copy (5–20GB), takes minutes to boot, and reserves hardware resources whether it needs them or not.

Era 3 — Containers. All Linux systems already share the same kernel. Containers isolate a process without duplicating the entire OS — lightweight, fast to start, and portable.

mental model A container is an isolated process that thinks it's the only thing running on its own private machine. It shares the host's kernel but has its own filesystem, network, and process list.

VM vs. Container

VMContainer
Size5–20 GB50–500 MB
Boot time2–5 minutes< 1 second
OSFull OS per VMShared kernel
IsolationHardware-levelProcess-level (namespaces)

The shipping container analogy

Before standardized shipping containers, loading cargo was chaos — every item needed different handling. The standardized container made the outside uniform so the inside could be anything. Docker does the same for software: your app, dependencies, and config get packaged into a standard unit that runs identically anywhere.

Installing on Windows — WSL2

Containers rely on two Linux kernel features — namespaces (isolation) and cgroups (resource limits) — that don't exist in the Windows kernel. Docker Desktop on Windows runs a lightweight Linux VM via WSL2 to provide them. Skipping WSL2 forces a fallback to Hyper-V, which requires Windows Pro/Enterprise and is heavier — or Docker simply doesn't run at all on Windows Home. 

# Enable WSL2 (PowerShell, as Administrator)
wsl --install

# Verify
wsl --list --verbose

# Verify Docker
docker --version
docker run hello-world

Image vs. container — the first real proof

Running docker run pulls an image (the blueprint) and creates a container (a running instance of it). Removing a container never removes the image — it's just the instance that disappears. 

docker run -d -p 8080:80 --name my-nginx nginx
docker stop my-nginx
docker rm my-nginx

docker images   # nginx is still here
docker ps -a    # container is gone
exit codes Exited (0) means the process finished on its own successfully. Exited (137) means it was killed by an external signal — e.g. docker stop.
