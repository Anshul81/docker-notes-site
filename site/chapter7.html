Docker Compose

Why Compose exists

Running a multi-container stack manually means several docker run commands, each with multiple flags, executed in a specific order. Compose describes the entire stack — services, networks, volumes — in one YAML file, brought up with a single command.

The full stack

services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

  flask-app:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      DB_HOST: postgres
      REDIS_HOST: redis

volumes:
  pgdata:

Service name = hostname

Compose automatically creates one network for the whole project and attaches every service to it. DNS resolution uses the service name (the YAML key, e.g. postgres) — not container_name, which is purely cosmetic and has no effect on networking.

depends_on ≠ ready

real gap depends_on alone only guarantees start order, not that the dependency has finished initializing. A container can "start" before its internal service (e.g. Postgres) is actually accepting connections — a race condition. Fixed with healthcheck + depends_on: condition: service_healthy, which waits for an actual readiness check to pass, not just for the process to launch.

Automatic DB init

The official postgres image auto-runs any .sql file placed in /docker-entrypoint-initdb.d/ — but only on a genuinely empty data directory, on first startup. Bind-mounting init.sql there means the schema is ready before the app ever makes its first request — no manual psql steps needed.

catch If the volume already contains any Postgres data (even from a failed earlier attempt), init is skipped entirely. Force a clean re-init with docker compose down -v before up.

--build vs. plain up

docker compose up only builds automatically the very first time. After that, it reuses whatever image is cached — code changes to app.py silently won't apply unless you explicitly rebuild: 

docker compose up --build

Note this is a completely separate concern from volume data — changing init.sql needs down -v + up, not --build, since init scripts aren't part of the image at all.

down vs. down -v

CommandContainersNetworkVolumes
docker compose downremovedremovedkept
docker compose down -vremovedremoveddeleted

Redis caching pattern

cached = redis_client.get('students_cache')
if cached:
    return {"students": json.loads(cached), "source": "cache"}

# ...query Postgres...

redis_client.setex('students_cache', 30, json.dumps(students))
return {"students": students, "source": "database"}

Check the cache first. On a miss, query Postgres and store the result with a 30-second expiry (setex). Verified the full lifecycle directly: first request → database, next several within 30s → cache, request after expiry → database again.

Project-scoped naming

Compose prefixes volumes and networks with the project folder name (chapter7_pgdata, chapter7_default) automatically — preventing two unrelated projects that both use a volume called pgdata from accidentally colliding or corrupting each other's data.

Compose command reference

docker compose up              # start (foreground)
docker compose up -d           # start (detached)
docker compose up --build      # rebuild images first
docker compose ps              # project-scoped container list
docker compose logs -f <svc>   # follow logs for one service
docker compose down            # stop + remove containers/network, KEEP volumes
docker compose down -v         # also DELETE volumes
docker compose restart <svc>   # restart one service
